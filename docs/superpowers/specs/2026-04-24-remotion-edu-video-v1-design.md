# Remotion Edu Video V1 Design

**Date:** 2026-04-24

**Status:** Draft approved for V1 scope, pending implementation planning

## 1. Goal

Build a V1 education video generation system on top of Remotion for the K12 tutoring market.

The system takes a math problem as text input and produces a 30 to 60 second explanation video with:

- step-by-step teaching script
- synthesized voice-over
- sentence-level subtitles
- reusable scene animations
- exported MP4 video

V1 is intentionally narrow:

- input only supports text
- subject only supports junior high math
- problem type only supports linear equations in one variable
- output only supports one fixed video ratio per deployment
- subtitles only support sentence-level timing

## 2. Non-Goals

The following are explicitly out of scope for V1:

- multi-subject support
- image OCR input
- free-form visual editing
- character-driven animation
- word-level subtitle alignment
- arbitrary problem type support
- direct LLM generation of Remotion props
- support for multiple output formats in the same pipeline

## 3. Product Definition

This product is not a general AI video generator.

It is an educational content generation engine with Remotion as the rendering layer.

Core product value:

1. Convert a text math problem into a stable teaching plan.
2. Map that teaching plan into reusable animation scenes.
3. Produce a teacher-like narrated explanation video with low manual effort.

Primary V1 user:

- tutoring institution content operator
- math teacher producing short explanation videos
- education product team validating automated question explanation workflows

Primary V1 use case:

- operator pastes a linear equation problem
- chooses duration and voice style
- system returns a generated explanation video and related assets

## 4. V1 Success Criteria

V1 is successful if it can reliably do the following:

1. Accept a text input such as `Solve equation: 2x + 3 = 11`
2. Generate a structured lesson plan with 4 to 6 explanation steps
3. Generate voice-over audio for each step
4. Generate sentence-level subtitles
5. Render a complete 45 second explanation video
6. Return the video together with structured job metadata

Operational targets for V1:

- at least 80 percent of supported inputs complete without manual correction
- end-to-end generation completes within a reasonable async job window
- lesson structure remains schema-valid after retries
- rendering remains stable for the supported template set

## 5. Product Constraints

To keep the V1 stable, the following constraints are mandatory:

- LLM only outputs teaching structure, never final render props
- Scene props are always generated by deterministic application code
- Video duration is derived from audio duration plus fixed timing rules
- Templates are fixed and reusable, not generated on the fly
- Problem type must be classified before lesson planning begins

## 6. System Architecture

The system is divided into six layers.

### 6.1 Input Understanding Layer

Responsibilities:

- receive user problem text
- normalize request options
- classify supported problem type
- reject unsupported inputs early

Input examples:

- problem text
- target duration
- student level
- narration style

Output:

- normalized `ProblemInput`
- parsed problem metadata

### 6.2 Teaching Planning Layer

Responsibilities:

- generate a teaching-friendly explanation plan
- decompose the solution into steps
- identify common mistakes
- define visual intent for each step

This is the product intelligence layer.

Output:

- `LessonPlan`

### 6.3 Template Mapping Layer

Responsibilities:

- transform `LessonPlan` into stable video scenes
- select the correct Remotion scene types
- generate deterministic props for each scene

Output:

- `VideoProject`
- `VideoScene[]`

### 6.4 Audio Layer

Responsibilities:

- synthesize voice-over per scene
- measure returned audio duration
- feed timing metadata back into the scene configuration

Output:

- scene audio files
- subtitle text
- duration metadata

### 6.5 Render Layer

Responsibilities:

- receive normalized composition props
- compose video scenes
- place subtitles and audio
- export MP4

This layer is implemented with Remotion and `@remotion/renderer`.

### 6.6 Orchestration Layer

Responsibilities:

- create and track jobs
- coordinate LLM, TTS, and render stages
- handle retries and failures
- persist outputs and metadata

## 7. Recommended Repository Structure

V1 should use a TypeScript monorepo with clean package boundaries.

```text
edu-video-gen/
|-- apps/
|   |-- web/
|   `-- api/
|-- packages/
|   |-- renderer/
|   |-- lesson-engine/
|   |-- tts-service/
|   |-- shared-types/
|   `-- job-runner/
|-- infra/
|   |-- docker/
|   `-- scripts/
`-- docs/
```

Module intent:

- `apps/web`: input form, task list, result preview
- `apps/api`: job creation and job query APIs
- `packages/renderer`: Remotion compositions, scenes, themes
- `packages/lesson-engine`: parsing, prompting, schema validation, mapping
- `packages/tts-service`: audio synthesis adapter
- `packages/shared-types`: canonical cross-package schemas
- `packages/job-runner`: async orchestration and pipeline execution

## 9. LLM Design

V1 should use a staged LLM pipeline instead of one-shot generation.

### Stage 1: Problem Parsing

Output:

- subject
- supported problem type
- difficulty
- knowledge point classification
- supportability flag

### Stage 2: Teaching Planning

Output:

- lesson goal
- explanation steps
- narration per step
- visual intent
- common mistake note

### Stage 3: Validation and Repair

Flow:

1. Validate against schema
2. If invalid, attempt structured repair
3. If still invalid, retry with narrower prompt
4. If still invalid, fail the job with explicit reason

Rules:

- model never emits final Remotion props
- model must stay inside step schema
- unsupported problems should fail early, not hallucinate a plan

## 10. Scene Library

V1 scene library includes exactly five scene types.

### 10.1 TitleScene

Purpose:

- show lesson title
- establish clean classroom branding

### 10.2 ProblemScene

Purpose:

- display original equation
- highlight known terms
- introduce the task

### 10.3 EquationTransformScene

Purpose:

- animate algebraic transformation
- highlight the operation being applied
- present one core step at a time

### 10.4 MistakeWarningScene

Purpose:

- show a common wrong move
- contrast with the correct operation

### 10.5 SummaryScene

Purpose:

- present the final answer
- summarize the solving method
- reinforce transfer to similar problems

## 11. Timing and Subtitle Strategy

V1 timing must stay simple and deterministic.

Rules:

- TTS is generated per scene or per lesson step
- returned audio duration becomes the base timing source
- scene duration equals audio duration plus small visual padding
- subtitles are sentence-level only
- internal animation timing is percentage-based

Example timing split for a step scene:

- 0 to 20 percent: show current equation
- 20 to 50 percent: highlight target term
- 50 to 80 percent: animate transformation
- 80 to 100 percent: settle on the result

This avoids fragile word-level alignment while keeping scenes paced correctly.

## 12. Render Boundary

Remotion should only receive a fully prepared render model.

Correct flow:

`ProblemInput -> ParsedProblem -> LessonPlan -> VideoProject -> Remotion inputProps -> renderMedia()`

Incorrect flow:

`ProblemInput -> Remotion component tries to infer lesson structure`

Design rule:

- Remotion is the rendering engine, not the lesson planner

Implementation implication:

- `selectComposition()` and `renderMedia()` should receive validated props
- the same input props must be used consistently during composition selection and rendering

## 13. Orchestration Flow

End-to-end V1 job flow:

1. User submits problem text and options
2. API creates async job record
3. Job runner parses and validates input
4. Lesson engine generates `ParsedProblem`
5. Lesson engine generates `LessonPlan`
6. Mapper builds `VideoScene[]`
7. TTS generates audio for each scene
8. Duration metadata is written back into scene config
9. Subtitle data is generated
10. Renderer builds final video through Remotion
11. System stores `mp4`, subtitles, and job metadata
12. API returns completed assets

## 14. API Surface

V1 only needs two external endpoints.

### `POST /jobs`

Purpose:

- create a generation job

Request body:

- `ProblemInput`

Response:

- `jobId`
- initial status

### `GET /jobs/:id`

Purpose:

- fetch current job state and outputs

Response:

- job status
- current stage
- error info if failed
- generated asset URLs if completed

Internal service interfaces:

- `parseProblem(input): Promise<ParsedProblem>`
- `planLesson(parsed): Promise<LessonPlan>`
- `mapLessonToScenes(plan): VideoScene[]`
- `synthesizeSceneAudio(scene): Promise<AudioMeta>`
- `renderProject(project): Promise<RenderArtifact>`

## 15. Deployment Direction

Recommended V1 backend stack:

- Node.js
- TypeScript
- Next.js or NestJS for API
- BullMQ and Redis for async jobs
- PostgreSQL for jobs and metadata
- object storage for audio and video assets

V1 deployment model:

- one API process
- one worker process
- one render-capable environment

Render should remain asynchronous from day one because video generation is not a request-response operation.

## 16. Failure Handling

### Input failures

- unsupported subject
- unsupported problem type
- malformed equation

Handling:

- fail early with structured reason

### LLM failures

- invalid schema output
- inconsistent step order
- unsupported reasoning path

Handling:

- schema validation
- repair pass
- bounded retry count

### TTS failures

- synthesis API timeout
- empty audio output

Handling:

- retry with same text
- fail job if audio remains unavailable

### Render failures

- invalid props
- missing audio asset
- Remotion render crash

Handling:

- validate render model before render
- persist failure stage and diagnostic message

### Week 1: Static Render Loop

Goal:

- prove that Remotion can render a linear equation explanation video from handwritten JSON

Deliverables:

- one composition

### Week 2: Text to LessonPlan

### Week 3: Audio and Timing

### Week 4: Job System and Web UI

## 18. Key Risks and Mitigations

### Risk 1: LLM output instability

Mitigation:

- narrow scope to one problem type
- use staged prompts
- enforce schema validation

### Risk 2: Algebra visuals become ad hoc

Mitigation:

- build one canonical equation transform scene
- keep props deterministic and typed

### Risk 3: Audio timing varies too much

Mitigation:

- derive scene duration from audio
- use percentage-based animation segments

### Risk 4: Render pipeline becomes slow or brittle

Mitigation:

- keep V1 videos short
- render asynchronously
- cap resolution and composition count

## 19. Final Recommendation

V1 should be built as a narrow, stable system with the following final scope:

- text input only
- junior high math only
- one-variable linear equations only
- 45 second explanation video target
- sentence-level subtitles
- deterministic scene mapping
- Remotion as render engine only

If this scope is delivered well, the next version can expand in this order:

1. more algebra problem types
2. image OCR input
3. geometry support
4. reusable teacher-facing content operations tools
